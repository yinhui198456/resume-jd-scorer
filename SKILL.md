# resume-jd-scorer

JD 简历匹配度评估 skill。输入 JD 和简历，输出结构化评分结果。

## 触发场景

用户说以下任一内容：
- "评估简历"、"简历评估"、"JD 匹配度"、"面试评估"
- `/resume-jd-scorer`
- 上传 JD 文件或简历文件（.docx/.pdf/.bin/图片）并要求评估
- "看看这个简历"、"再看一下"（配合已上传的文件）

## 文件上传处理（重要）

**用户上传文件后，文件存放在 `.codepilot-uploads/` 目录，扩展名为 `.bin`。**

**第一步：先搜索最近的上传文件**

```bash
python3 <skill_dir>/scripts/find_uploads.py --type resume --with-name
```

这会列出最近的简历文件并自动提取候选人姓名，例如：
```
2026-06-25 13:41  426KB  application/pdf  王洁玉  /opt/personal-agent-workspace/.codepilot-uploads/xxx.bin
```

**第二步：确认用户要评估的是哪份简历**

- 如果只有一份最新简历 → 直接使用
- 如果有多份 → 询问用户确认哪一份
- **如果搜索结果为空** → 提示用户先上传简历文件，不要凭空假设

**第三步：解析文件内容**

```bash
python3 <skill_dir>/scripts/parse_file.py <文件路径>
```

`parse_file.py` 会自动检测 MIME type（不依赖扩展名），支持：
- `application/pdf` → PDF 文本提取
- `application/zip` → DOCX 解析
- `image/png` / `image/jpeg` / `image/webp` → OCR 识别
- `text/plain` → 直接读取

> **禁止行为：** 如果没有找到上传文件，不要编造简历内容。直接提示"未找到上传的简历文件，请先上传"。

## 交互流程

### 第一步：收集 JD

**JD 持久化：首次上传的 JD 会自动保存，后续评估默认复用。**

**1. 检查是否有已保存的 JD：**

```bash
python3 <skill_dir>/scripts/save_jd.py has
```

- **如果输出 "已保存 | ..."** → 加载已保存的 JD：
  ```bash
  python3 <skill_dir>/scripts/save_jd.py load
  ```
  读取输出作为 JD 文本。向用户确认："使用已保存的 JD（来源：xxx），还是需要更换？"

- **如果输出 "未保存 JD"** → 按以下优先级收集 JD：
  1. 用户消息中是否直接提供了 JD 文本？→ 使用并保存
  2. 用户是否上传了 JD 文件？→ 用 `find_uploads.py --type jd` 搜索，解析后保存
  3. 都没有 → 提示用户提供 JD

**保存 JD（收到 JD 文本后执行）：**

```bash
python3 <skill_dir>/scripts/save_jd.py save "<JD文本>" "<来源标识>"
```

来源标识示例：`file:ai_infra.pdf`、`user_input`、`clipboard`

### 第二步：收集 CV/简历

**优先使用 find_uploads.py 搜索用户上传的文件。**

运行：
```bash
python3 <skill_dir>/scripts/find_uploads.py --type resume --with-name
```

- **如果找到文件** → 选择最新的一份（或让用户确认），用 parse_file.py 解析
- **如果未找到** → 提示用户："未找到上传的简历文件，请先上传 PDF/Word/图片格式的简历"

### 第三步：执行评估

将 JD 和简历文本写入临时文件，运行评估脚本：
```bash
JD_TEXT="<jd文本>" CV_TEXT="<cv文本>" python3 <skill_dir>/scripts/evaluate.py
```

脚本调用 LLM 进行评估，输出 JSON 格式结果。

### 第四步：展示结果

#### A. 候选人信息汇总（仅最终得分 ≥ 50 时输出）

按以下 Tab 分隔格式输出（不要使用 Markdown 表格）：

```
姓名	录入时间	联系方式	学历	工作年限	硬性	技能	经验	潜力	可信系数	合计
吕蜜	2026/6/25	18918078977 / milvlinux@163.com	大专+本科（华中农业大学计算机）	10年+（简历标注1年有误）	38	28	18	7	0.6	55
```

**字段说明**：
- 硬性 = 硬性要求匹配度得分（满分40）
- 技能 = 技能匹配度得分（满分30）
- 经验 = 经验匹配度得分（满分20）
- 潜力 = 潜力/加分项得分（满分10）
- 可信系数 = 可信度乘数（1.0/0.9/0.75/0.6/0.5）
- 合计 = 最终得分（round(硬性+技能+经验+潜力) × 可信系数）

**评级标准**：≥75 INTERVIEW | 50-74 BACKUP | <50 REJECT

#### B. 面试题库（15-20 道，结合 JD + CV 生成）

题库必须覆盖以下四个类别，每个类别 3-5 题：

**验证类（针对简历疑点）**
- 针对简历中可疑的描述（年限矛盾、"精通"无支撑、项目模板化等）
- 要求候选人提供具体场景、数据、决策过程

**技术类（覆盖 JD 各项技术要求）**
- 逐一覆盖 JD 中的技术栈要求
- 结合候选人简历中的职责描述，问实际操作深度
- 例：JD 要求 K8s 基础排查 → 问具体排查命令和案例

**场景类（针对岗位实际工作）**
- 驻场交付场景：客户沟通、应急响应、需求变更处理
- 日常运维场景：巡检异常处理、故障排查 SOP
- 新技术上手：未接触过的产品如何快速学习

**综合能力类（必问）**
- 沟通表达：如何向非技术人员解释技术问题
- 出差适应：能否接受频繁出差驻场
- 对客能力：客户现场的压力处理和情绪管理
- 工作强度：能否接受项目上线期的加班节奏
- 文档汇报：技术方案编写、故障复盘、周报月报能力

#### C. 详细评估（可选展开）

```

═══════════════════════

───────────────

技能匹配：27 / 30 — 熟悉 Spring Boot/MySQL/Redis
经验匹配：15 / 20 — 电商订单和支付经验
潜力加分：8  / 10 — 大厂背景 + 开源贡献

💪 优势
• 5年Java经验
• 有大厂背景

️ 短板
• Kafka实战不足

🚩 红旗信号（如有）
• "精通K8s"但无集群规模/故障排查案例支撑
• 项目时间线有重叠（2023.01-2024.06 与 2023.06-至今）

## 评分规则

| 维度 | 满分 | 说明 |
|------|------|------|
| 硬性要求匹配度 | 40 | 学历、年限、语言能力等 |
| 技能匹配度 | 30 | 硬技能/工具/框架覆盖度 |
| 经验匹配度 | 20 | 过往业务场景与 JD 职责匹配度 |
| 潜力/加分项 | 10 | 开源/大厂背景/跨领域能力 |

**️ 真实性验证（不单独计分，但影响总分判断）**

评估时 LLM 必须主动识别以下红旗信号，并在 weaknesses 中标注：

| 红旗类型 | 识别方法 | 排除项（不算红旗） |
|---------|---------|---------|
| 年限与内容不符 | 自称年限 vs 实际描述的深度 **排除**：简历头部年限明显笔误（如写"1年"但履历完整覆盖10年） | 明显笔误不算红旗，以实际履历时间线为准 |
| 技能描述过度 | 滥用"精通"但缺乏对应项目支撑 | "精通"一词本身不构成红旗，需看是否有对应项目/场景支撑 |
| 项目真实性存疑 | 项目描述像培训材料/网课作业（通用项目名"电商平台/管理系统"、缺乏具体业务指标、多项目模板化重复） | **排除**：PDF解析产生的乱码水印字符串（如 `dd12b1e34f370bd21HF92Ny8FVpXxIq3UfqcWOelmv7WPxBh`），这是解析噪声 |
| 职责范围夸张 | 初级岗位写高级职责 | **排除**：团队项目中候选人仅负责其中一部分，以"个人职责"描述为准，不苛求覆盖整个项目 |
| 时间线矛盾 | 教育/工作时间重叠或倒置 | 在校实习/兼职不算矛盾 |
| 技术栈跳跃 | 缺乏过渡直接跳新技术 | 自学/培训课程作为过渡是合理的 |
| 培训痕迹 | 简历格式/用语像培训班模板 | 模板化格式本身不致命，关键看项目内容是否真实可验证 |

**追问设计原则：**
- 每个 flagged 的技能/项目必须有 **验证性追问**（不是泛泛的技术问题）
- 追问应要求候选人提供 **具体场景、数据、决策过程**，而非概念性回答
- 例：不要问"K8s 有哪些组件"，要问"你管理的集群规模多大？遇到过最棘手的 Pod 调度问题是什么，如何排查的？"

### 建议阈值

| 总分 | 建议 | 含义 |
|------|------|------|
| ≥ 75 | 建议面试 | 硬性要求基本满足，匹配度高 |
| 50-74 | 备选 | 部分维度有缺口但有一定可塑性 |
| < 50 | 不建议 | 硬性要求缺失或整体匹配度过低 |

## 支持文件

| 文件 | 路径 | 用途 |
|------|------|------|
| `scripts/save_jd.py` | `<skill_dir>/scripts/save_jd.py` | JD 持久化：保存/加载/检查已保存的 JD |
| `scripts/find_uploads.py` | `<skill_dir>/scripts/find_uploads.py` | 搜索最近上传的简历文件，自动提取候选人姓名 |
| `scripts/parse_file.py` | `<skill_dir>/scripts/parse_file.py` | 解析 PDF/DOCX/TXT/图片 提取纯文本 |
| `scripts/evaluate.py` | `<skill_dir>/scripts/evaluate.py` | LLM 评估核心逻辑（调用 Anthropic API） |
| `data/current_jd.json` | `<skill_dir>/data/current_jd.json` | 最近保存的 JD 数据（自动创建） |

## 环境变量

评估脚本需要以下环境变量（由 Claude Code 会话自动提供）：
- `ANTHROPIC_AUTH_TOKEN` — API 密钥
- `ANTHROPIC_BASE_URL` — API 代理地址
- `ANTHROPIC_MODEL` — 模型名称（默认 `qwen3.6-plus`）

## 错误处理

| 错误 | 处理 |
|------|------|
| 未找到上传文件 | 提示"未找到上传的简历文件，请先上传"，**禁止编造内容** |
| 无已保存 JD 且用户未提供 | 提示"请先上传 JD 文件或粘贴 JD 文本" |
| JD 文件损坏/解析失败 | 提示"JD 文件无法解析，请重新上传" |
| 文件格式不支持 | 提示仅支持 PDF/DOCX/TXT/图片格式 |
| 文件不存在 | 提示路径不正确 |
| LLM 调用失败 | 提示"评估服务暂时不可用"，不暴露堆栈 |
| 空输入 | 在收集阶段拦截，不允许空 JD 或空简历进入评估 |
| 多份简历时 | 列出候选人姓名让用户确认，不要随机选 |
| 用户要求更换 JD | 清除旧 JD（重新保存新 JD 即自动覆盖） |
