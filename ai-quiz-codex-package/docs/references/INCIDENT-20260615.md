# 刷题事故复盘 — 2026-06-15 M04 Context工程

## 事故概述

今日用户在 M04 模块刷题时，连续出现以下问题：
1. Q49 反复出现 5-6 次
2. Q50 同样反复出现
3. **Q52 严重错位**：LLM 凭空展示了一道"全局上下文广播"题（不在题库），用户作答后被判题到实际 Q52（95% 阈值题）
4. 飞书 thread 引用导致 LLM 误判当前题号

## 根本原因

### 原因 1（最严重）：LLM 编造题目
- LLM 在压力下凭上下文/记忆捏造了"全局上下文广播"题
- 违反 quiz-bot 铁律：禁止凭记忆编题
- 后果：用户以为自己学了这道题，实际题库中根本不存在

### 原因 2：飞书 thread 引用导致 QID 错位
- 用户每次回复飞书默认引用历史 Q49 消息
- LLM 没从 state 读 current_qid，凭"刚展示的题"判断
- 答错/答对会被记到错误 QID

### 原因 3：LLM 跳过 answer 步骤
- 中间多次展示题目后没有真正调用 `quiz_bot.py answer`
- 导致 tracking 未更新，next 继续返回同一题

### 原因 4：snapshot_mismatch 静默放行
- P0 修复把 snapshot_mismatch 降级为警告
- LLM 没有主动重出题，导致 hash 不一致的题目被错位判题

## 错位影响

| QID | 用户答的 | 实际题库题 | 学习有效性 |
|-----|---------|-----------|----------|
| Q49 | C | 上下文无状态 → 答案 A | ✅ 真实判错 |

| Q53 | 未答 | Context vs Prompt | - |

## 改进 SOP

### SOP-1：答题前置三查（强制）
收到用户 A/B/C/D 回复时：
1. 读 `tracking/quiz_bot_state.json` 的 `current_qid`
2. 校验 state.current_qid 与 answer qid 一致
3. 检查飞书 "Replying to:" 引用是否与当前题对应

### SOP-2：禁止凭记忆编题（已有铁律，强化执行）
- 每道题必须来自 `quiz_bot.py next` 输出
- LLM 不能凭上下文/记忆补全或重写题目
- 出题脚本输出是唯一事实源

### SOP-3：hash mismatch 必须重出题
- `snapshot_mismatch` 命中时，LLM 必须**主动调用 `next` 重新出题**
- 不是只记录警告就放行

### SOP-4：飞书 thread 引用时显式确认
- 收到 "Replying to: Q49" 但 state 是 Q51 时
- 在 answer 前先向用户确认

### SOP-5：每答完一题主动 sync
- 答完一题后调 `status` 一次（每 3 题）
- 防止 LLM 凭感觉判断进度

### SOP-6：answer 命令前校验 QID 一致性
- `answer` 命令新增参数校验：state.current_qid != answer qid → 拒绝
- 防止错位判题
